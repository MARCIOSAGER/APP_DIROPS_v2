<objective>
Establish the safe foundation for caching: clear the query cache on logout (preventing cross-user data leakage), guard TOKEN_REFRESHED against ghost re-renders, and add a 15-second fetch timeout to prevent indefinite hangs.

Purpose: These are correctness fixes, not optimizations. Without cache clearing on logout, a shared terminal can show one user's data to the next user. Without the TOKEN_REFRESHED guard, token refresh mid-session causes a visible re-render. Without a timeout, a slow Supabase connection hangs forever with a spinner.

Output: Modified supabaseClient.js, query-client.js, and AuthContext.jsx. No new files.
</objective>

**supabaseClient.js:** Add a `global.fetch` override to the createClient options that wraps every Supabase request with `AbortSignal.timeout(15000)`. Place it after the existing `auth` block. The `lock` function on the auth options must be preserved exactly as-is.

```js
export const supabase = createClient(supabaseUrl, supabaseAnonKey, {
  auth: {
    persistSession: true,
    autoRefreshToken: true,
    detectSessionInUrl: true,
    lock: async (_name, _acquireTimeout, fn) => await fn(),
```

**Change 3 — Add cache clear to SIGNED_OUT handler AND guard TOKEN_REFRESHED:**
The current `onAuthStateChange` handler section (around lines 118-128):
```js
if (event === 'SIGNED_IN' && session?.user) {
  await loadUserProfile(session.user);
  setIsLoadingAuth(false);
} else if (event === 'SIGNED_OUT') {
  setUser(null);
  setIsAuthenticated(false);
```

Replace with:
```js
if (event === 'SIGNED_IN' && session?.user) {
  await loadUserProfile(session.user);
  setIsLoadingAuth(false);
} else if (event === 'SIGNED_OUT') {
  queryClientInstance.clear();
  setUser(null);
  setIsAuthenticated(false);
  setIsLoadingAuth(false);
} else if (event === 'TOKEN_REFRESHED' && session?.user) {
  // Guard: only reload profile if the user identity actually changed.
  // TOKEN_REFRESHED fires every ~55 minutes (Supabase token lifetime).
  // If the user.id is the same, the profile is unchanged — skip the DB call
  // to prevent a ghost re-render mid-session.
  if (session.user.id !== user?.id) {
    await loadUserProfile(session.user);
  }
}
```

**IMPORTANT:** The `user` referenced in the TOKEN_REFRESHED guard refers to the React state variable `user` (defined at line 8: `const [user, setUser] = useState(null)`). This is in scope inside the useEffect because the useEffect depends on `loadUserProfile` (which captures `user` via closure from the state). However, the useEffect dependency array is `[loadUserProfile]` only — this means `user` in the closure may be stale if the component re-renders. To ensure the guard reads the current value, use a ref pattern OR simply verify: since `user` is set during SIGNED_IN before TOKEN_REFRESHED can fire, and the same user re-authenticating keeps the same `user.id`, the closure value is sufficient for this guard. Do NOT add `user` to the useEffect dependency array (it would cause infinite loops). The closure is safe here.
